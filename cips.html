CIPs

Community Improvement Proposals (CIPs) describe standards for the Genesis platform, including core protocol specifications, client APIs, and contract standards.

Contributing

First review CIP-1. Then clone the repository and add your CIP to it. There is a template CIP here. Then submit a Pull Request to Genesis's CIPs repository.

CIP status terms

  • Draft - an CIP that is open for consideration and is undergoing rapid iteration and changes.
  • Last Call - an CIP that is done with its initial iteration and ready for review by a wide audience.
  • Accepted - a core CIP that has been in Last Call for at least 2 weeks and any technical changes that were requested have been addressed by the author. The process for Core Devs to decide whether to encode an CIP into their clients as part of a hard fork is not part of the CIP process. If such a decision is made, the CIP will move to final.
  • Final (non-Core) - an CIP that has been in Last Call for at least 2 weeks and any technical changes that were requested have been addressed by the author.
  • Final (Core) - an CIP that the Core Devs have decided to implement and release in a future hard fork or has already bZZeen released in a hard fork.

CIP Types

Community Improvement Proposals describe any change that affects most or all Genesis implementations, such as a change to the the community guidelines, proposed application standards/conventions, or any change or addition that affects the interoperability of applications using Genesis. Furthermore CIPs can be broken down into the following categories.

Core ({{site.pages|where:"type","Core"|size}})

Improvements to the core community documents, requiring changes to a core community document, or adding a new core community document. The core commnuity documents can be found here: FIXME.

Governance ({{site.pages|where:"type","Governance"|size}})

Includes improvements around governance for Genesis, including governance contracts, community organization, and foundation. They may propose an implementation, but not to Genesis's codebase; they often require community consensus; hey are more than recommendations, and users are typically not free to ignore them. Examples include procedures, guidelines, changes to the decision-making process, and changes to the tools or environment used in Genesis development.

GRC ({{site.pages|where:"type","GRC"|size}})

GRC (Genesis Request for Comment) are Application-level standards and conventions, including contract standards such as token standards for in-game items, software development standards, URI schemes, API standards, etc.

Game Dev ({{site.pages|where:"type","GameDev"|size}})

Describes a Genesis game design or development issue, across game mechanics, lore, art, etc. All Game Dev CIPs of this type are further split into sub-categories:

Gameplay ({{site.pages|where:"type","GameDev"|where:"category","Gameplay"|size}})

Includes anything related to the gameplay of Genesis, whether it be core gameplay, or gameplay specific to a world or minigame.

Art ({{site.pages|where:"type","GameDev"|where:"category","Art"|size}})

Includes anything related to the art, graphics, or visual design of Genesis, whether it be environments, enemies, skins, or development tools & standards.

Lore ({{site.pages|where:"type","GameDev"|where:"category","Lore"|size}})

Includes anything related to the lore, back-story, of Genesis, or its worlds.

Economy ({{site.pages|where:"type","GameDev"|where:"category","Economy"|size}})

Includes anything related to the in-game economy of Genesis. Tokenomic discussions should be done as a GRC instead.

Software ({{site.pages|where:"type","GameDev"|where:"category","Software"|size}})

Includes anything related to the software required to run the game of Genesis.
